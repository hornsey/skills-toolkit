# brainstorming

> 结构化头脑风暴技能包 — 在动手实现任何创意之前，先把设计想清楚

## 🎯 痛点 & 问题

在日常开发中，我们经常会遇到以下困扰：

- **需求不清晰就动手**：还没完全理解用户意图，就开始写代码，导致返工
- **设计被跳过**：觉得"这是一个小改动，不需要设计"，结果埋下隐患
- **思路混乱**：头脑中有多种方案，但不知道如何系统地比较和选择
- **实现过程缺乏指引**：设计完成后，不知道下一步该做什么
- **文档缺失**：做完就忘，没有留下可追溯的设计决策记录

**brainstorming Skill 解决的核心问题**：强制在创意与实现之间插入结构化的设计对话环节，确保每一次开发都从清晰的需求理解开始，形成可追溯的设计文档。

## ⚙️ 原理

### 底层机制

brainstorming 是 `obra/superpowers` 技能集合中的核心环节，基于以下机制设计：

1. **渐进式需求探索**：通过一问一答的对话形式，逐步澄清用户意图和约束条件
2. **设计文档生成**：将讨论结果写入 `docs/superpowers/specs/YYYY-MM-DD-<topic>-design.md` 格式的文档
3. **强制用户审核门控**：设计完成后必须获得用户确认，才能进入下一步
4. **技能链式调用**：brainstorming 完成后只能调用 `writing-plans` 技能，禁止跳过环节直接实现

### 依赖工具/接口

- **SKILL.md 格式**：符合 Agent Skills 开放标准（Anthropic 提出，现被 OpenAI、Cursor、GitHub Copilot 等广泛采用）
- **skills.sh 兼容**：可与 Vercel 的 skills.sh 技能市场集成
- **支持客户端**：Claude Code、OpenAI Codex CLI、Cursor、GitHub Copilot 等

### 核心逻辑

```
用户请求 → 触发 brainstorming → 理解想法 → 探索方案 → 呈现设计 → 用户审核 → 写入设计文档 → 调用 writing-plans
```

**关键约束**：
- 禁止在未完成设计的情况下直接调用 `frontend-design`、`mcp-builder` 或其他实现技能
- 即使是"简单"任务（如修改配置），也必须完成设计流程
- 设计文档必须包含足够信息，供后续实现参考

### 使用示例

**场景 1：计划一个新功能**

```
用户：我想为我们的 SaaS 添加一个订阅管理功能，让用户可以升级、降级或取消订阅。

AI（触发 brainstorming）：
我需要先理解一些细节，才能帮你设计这个功能...
[开始提问：订阅层级如何设置？是否需要试用期？计费周期是什么？支付方式有哪些？...]
```

**场景 2：开发一个组件**

```
用户：帮我加一个导出按钮

AI（触发 brainstorming）：
"导出按钮"这个需求我可以帮你实现，但在开始之前，我想确认几个问题...
[开始提问：导出的格式是什么？CSV、Excel 还是 PDF？数据来源是哪些？有没有行数限制？...]
```

**场景 3：重构现有代码**

```
用户：我觉得这个模块的代码太乱了，帮我重构一下

AI（触发 brainstorming）：
好的，让我先理解一下现状...
[开始提问：主要问题是什么？性能、可读性还是扩展性？有没有明确的性能指标需要保持？...]
```

### 工作流程

1. **理解想法**：AI 通过提问了解你要做什么、解决什么问题
2. **探索方案**：讨论实现思路、权衡利弊
3. **呈现设计**：AI 整理成结构化的设计文档
4. **审核门控**：用户确认设计后，才能继续

### 视觉辅助（可选）

brainstorming 支持浏览器端的视觉辅助工具，用于展示mockups、流程图等。AI 会在需要时询问你是否启用。

## ✅ 适用场景

- **新功能开发**：任何需要添加新功能、组件或服务的时候
- **代码重构**：需要修改现有代码结构之前
- **配置变更**：看似简单的配置修改（实际上往往涉及复杂的依赖关系）
- **技术选型**：需要在多种技术方案中做选择时
- **API 设计**：设计新的接口或修改现有接口
- **用户体验设计**：涉及 UI/UX 变动的任何修改
- **多人协作项目**：需要明确记录设计决策供团队参考

## ❌ 不适用场景

- **紧急修复**：生产环境紧急 bug 需要立即修复，此时不应强求完整的设计流程（但事后应补充文档）
- **简单查询**：纯阅读代码、理解逻辑，不需要设计
- **已冻结的项目**：需求明确、代码稳定，只需要例行维护
- **实验性探索**：快速验证某个想法的可行性，此时设计可能成为阻碍

## ⚠️ 局限 & 风险

1. **可能增加认知负担**：对于非常简单的任务，严格走流程可能显得繁琐
2. **依赖用户配合**：如果用户不认真回答问题，设计质量会受影响
3. **设计文档维护成本**：设计文档需要与实现保持同步，否则会过时
4. **技能冲突**：AI 可能被其他技能"劫持"，跳过 brainstorming 直接实现
5. **上下文窗口**：复杂项目的设计讨论可能消耗大量上下文空间

**缓解建议**：
- 对于真正简单的任务，可以缩短设计文档，但不要完全跳过
- 保持设计文档的简洁性，只记录关键决策
- 如果经常需要跳过 brainstorming，考虑是否任务拆分不够细
