## 🎯 痛点 & 问题

如果你每天都要追踪 AI 和技术动态，最常见的问题并不是“找不到信息”，而是：

- **信息源太多**：关注的博客、RSS、社区越多，越难稳定浏览完
- **噪声比例高**：很多更新价值不大，真正值得读的内容容易被淹没
- **缺少中文整理**：英文博客和技术站点更新快，但中文团队同步整理成本高
- **日报制作费时**：手工筛选、翻译、归类和排版非常耗时
- **缺少可视化概览**：只有链接列表，难快速把握当天重点与趋势

`AI Daily Digest` 的作用，是把“抓取 → 筛选 → 翻译 → 分类 → 摘要 → 可视化”整合成一份日报生成流程。

## ⚙️ 原理

根据官方 SKILL.md 与仓库说明，这个 Skill 的基础思路是：

### 核心机制

1. **聚合高质量技术博客 RSS**
   - 以 Karpathy 推荐的 90 个热门技术博客为核心信息源
   - 适合关注 AI、工程、研究与工具动态

2. **AI 评分筛选**
   - 不是简单罗列当天所有文章
   - 而是借助模型对文章进行相关性和价值判断，筛出更值得读的内容

3. **结构化日报输出**
   - 生成 Markdown 结果
   - 包含中文标题、分类分组、趋势亮点等结构化内容

4. **可视化增强**
   - 仓库说明中提到可生成 Mermaid 图表与标签云
   - 让日报不仅可读，也适合直接转发或二次加工

### 脚本与配置

| 组件 | 作用 |
|------|------|
| `scripts/digest.ts` | 主脚本，负责抓取 RSS、AI 评分与摘要生成 |
| `~/.hn-daily-digest/config.json` | 配置持久化文件 |
| `/digest` | 主要触发命令 |
| Gemini / OpenAI 兼容 API | 作为 AI 评分与整理能力来源 |

### 工作流

```text
RSS 抓取 → 过滤新文章 → AI 评分筛选 → 中文整理与分类
→ 生成 Markdown 日报 → 附带图表 / 标签云等可视化摘要
```

## 📖 使用指南

### 安装方式

```bash
npx skills add https://github.com/vigorx777/ai-daily-digest --skill ai-daily-digest
```

### 基础触发方式

```text
/digest
```

Skill 通常会围绕以下问题做引导：

- 是否使用已保存配置
- 关注哪些来源或分类
- 是否生成中文标题或摘要
- 输出文件放在哪里

### 使用示例

#### 场景 1：生成今日 AI 日报

```text
用户：/digest
AI：我会读取已保存配置（如果存在），抓取最新博客更新，筛选高价值文章，并生成一份结构化日报。
```

#### 场景 2：技术团队晨报

```text
用户：帮我做一份今天适合工程团队阅读的 AI / 开发日报。
AI：我会偏向工程与工具类文章，减少泛资讯内容，并输出更适合内部转发的 Markdown 摘要。
```

#### 场景 3：内容选题池

```text
用户：给我一份可以作为公众号选题的 AI 资讯精选。
AI：我会把抓取内容按主题整理，优先保留有讨论价值和传播潜力的文章。
```

## ✅ 适用场景

- **团队晨报 / 周报**：为研发、产品、投资或内容团队生成稳定的阅读摘要
- **AI 行业跟踪**：持续关注模型、工具、工程实践和研究动态
- **内容创作选题**：从英文技术博客中筛选值得二次创作的话题
- **个人信息管理**：减少每天逐站点刷 RSS 的成本
- **知识型社群运营**：为社群沉淀高质量日报素材

---

## ❌ 不适用场景

- **需要秒级实时快讯**：它更像日报生成器，不是即时新闻推送系统
- **只关注单一中文媒体**：如果信息源很单一，这套流程可能偏重
- **完全离线环境**：需要联网抓 RSS，也通常需要模型 API
- **不接受 AI 评分干预**：若必须保留所有原始文章，Skill 的筛选机制反而会削弱完整性

---

## ⚠️ 局限 & 风险

1. **信息源偏差**：来源集合再大，也可能偏向特定技术社区或英文博客圈
2. **AI 筛选主观性**：评分模型会影响保留哪些文章，不等于绝对客观
3. **翻译与摘要误差**：中文标题或摘要可能与原文细节存在偏差
4. **API 成本**：高频生成日报会增加模型调用成本
5. **图表依赖展示环境**：Mermaid 与标签云在不同渲染环境中表现可能不同
