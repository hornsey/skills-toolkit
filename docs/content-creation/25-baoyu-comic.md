## 🎯 痛点 & 问题

把知识内容做成漫画，难点通常不在“会不会画”，而在下面这些环节：

- **知识内容不易传播**：纯文字教程、科普或人物故事阅读门槛高，转发率有限
- **分镜设计困难**：即使有主题，也很难拆成有节奏的画面和对白
- **风格不统一**：人物设定、配色、画风和叙事语气常常前后不一致
- **图片生成缺乏流程**：单张图好生成，但连续多页漫画很难保持角色与叙事连贯
- **教育内容容易变枯燥**：知识点太硬，会导致“有信息、没阅读欲望”

`baoyu-comic` 的核心价值，是把“主题理解 → 漫画脚本 → 分镜结构 → 图片生成后端选择”打通，让 AI 能协助产出更像作品、而不是几张随机插画。

## ⚙️ 原理

根据 `baoyu-comic` 的 SKILL.md，这个 Skill 是一个“知识漫画生成工作流”，重点不只是画图，而是先组织内容与叙事。

### 核心机制

1. **知识内容转漫画结构**
   - 适合把教程、传记、解释型内容、教育内容转成漫画叙事
   - 强调“知识点如何讲出来”，而不是只生成漂亮插图

2. **风格 × 语气组合**
   - Skill 支持按不同艺术风格和叙事口吻产出内容
   - 适合做“严肃科普版”“轻松故事版”“人物传记版”等不同表达

3. **多后端图像生成决策**
   - 文档中明确了图像生成后端的选择顺序
   - 优先使用运行时原生图像工具，其次才是外部或仓库内后端
   - 这意味着 Skill 会根据当前 Agent 环境动态选择最合适的制图能力

4. **面板级输出思路**
   - 强调分镜、面板布局、台词与说明文字
   - 更适合连续页漫画，而不是只产出一张封面图

### 依赖工具 / 接口

| 组件 | 作用 |
|------|------|
| 兼容 SKILL.md 的 Agent | 负责理解主题、拆解脚本与调度生成 |
| 图像生成后端 | 用于真正渲染漫画画面 |
| 仓库内参考文档 | 提供风格、流程与后端分支说明 |
| 可选本地/云端模型 | 作为不同运行时中的绘图能力来源 |

### 工作流

```text
原始主题 / 知识点 → 提炼目标受众与叙事方式 → 生成漫画结构与分镜
→ 选择图像生成后端 → 逐页/逐格产出 → 统一风格并校正内容
```

### 使用示例

#### 场景 1：科普漫画

```text
用户：把“为什么会有潮汐”做成 8 格知识漫画。
AI：我会先确定目标读者和讲解语气，再拆成分镜、对白和画面提示词，最后统一输出漫画草案。
```

#### 场景 2：人物故事漫画

```text
用户：把特斯拉和爱迪生的交流电之争画成一组漫画。
AI：我会按人物冲突线索拆解故事，突出关键事件和对话，让内容更适合连载式阅读。
```

#### 场景 3：教程可视化

```text
用户：把“如何写 OKR”做成适合新员工培训的漫画。
AI：我会先把抽象概念转成具体场景，再给出角色、分镜和文案建议，避免只剩概念堆砌。
```

### 提示词建议

提供以下信息，效果会更稳定：

- 目标读者是谁
- 需要几格 / 几页
- 希望的风格（科普、幽默、人物传记、极简信息图等）
- 是否要求中英双语
- 是要完整成品，还是只先出脚本与分镜

## ✅ 适用场景

- **知识科普传播**：把复杂知识压缩成适合社媒传播的漫画内容
- **培训材料可视化**：把企业内训、教育内容变得更易理解
- **人物传记改编**：把创业者故事、历史人物或案例故事做成漫画
- **教程拆解**：将步骤型教程转成更轻量的图文叙事
- **内容再利用**：把长文、播客、课程摘要改造成新的传播形态

---

## ❌ 不适用场景

- **追求专业漫画工业流水线**：商业出版级漫画仍需要专业编剧、主笔和后期团队
- **只想生成单张海报**：如果目标只是单图海报，通用图片生成 Skill 更直接
- **事实不能改写的严肃材料**：法律、医疗、学术原文不适合做过度戏剧化改编
- **完全没有主题边界**：如果既没有目标受众也没有内容范围，漫画化会失去焦点

---

## ⚠️ 局限 & 风险

1. **角色一致性问题**：多页漫画在不同生成批次中容易出现人物形象漂移
2. **知识失真风险**：为了叙事性做简化时，可能损失部分严谨性
3. **成本与时长**：如果逐格生成高质量图片，调用成本和制作周期都不低
4. **后端差异明显**：不同运行时或模型的出图风格、稳定性差异较大
5. **版权与素材合规**：若引用现有 IP、照片或人物肖像，需要额外确认使用权限
