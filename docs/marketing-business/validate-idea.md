# validate-idea

> 创业想法验证 Skill —— 在真正投入开发、招聘或投放之前，先验证是否有人愿意为你的想法买单

**Source:** [GitHub - slavingia/skills](https://github.com/slavingia/skills)

**Methodology:** 《The Minimalist Entrepreneur》 | **Platforms:** Claude Code / 兼容 SKILL.md 的 Agent 运行时

---

## 🎯 痛点 & 问题

创业项目失败，很多时候不是因为“做得不够努力”，而是因为一开始就验证错了方向：

- **以为自己发现了需求，其实只是自己的想象**
- **先做产品，后找用户**，结果几个月后才发现没人愿意买单
- **把“有人说不错”误当成验证通过**
- **不知道应该验证什么**：是用户、痛点、解决方案，还是付费意愿？

`validate-idea` 的核心目标，就是把“模糊想法”变成一个可以被现实检验的创业假设。

---

## ⚙️ 原理

该 Skill 基于 Sahil Lavingia 的《The Minimalist Entrepreneur》方法论，强调 **先验证，再建设**。

### 核心机制

1. **社区优先**
   - 先找到一群真实存在、愿意交流的人
   - 没有社区，就很难持续听到真实问题

2. **问题优先于方案**
   - 先确认问题是否真实、高频、值得解决
   - 不急着证明自己的方案多聪明

3. **手工验证优先于产品化**
   - 先用人工交付、访谈、页面测试或轻服务验证需求
   - 不要一开始就把大量时间投入到系统开发

4. **付费意愿才是强验证**
   - “觉得不错”不等于会付费
   - 能否拿到钱，是最有力量的验证信号之一

### 典型四步法

| 步骤 | 核心问题 |
|------|----------|
| **社区验证** | 这群人是否真实存在、可持续接触？ |
| **问题验证** | 他们是否真的有这个痛点？频率和强度如何？ |
| **解决方案验证** | 他们是否愿意尝试你提供的方式？ |
| **付费验证** | 他们是否愿意为结果付费？ |

## 📖 使用指南

### 安装方式

```bash
npx skills add https://github.com/slavingia/skills --skill validate-idea
```

### 常见触发方式

```text
请使用 validate-idea，帮我验证一个面向独立开发者的部署平台想法
```

```text
我想做一款 AI 写作工具，先别让我开发，先帮我判断值不值得做
```

```text
帮我设计一个最小成本的创业验证方案
```

### 使用示例

#### 场景 1：从想法到验证计划

```text
用户：我想做一款面向播客作者的 AI 剪辑工具。
AI：我会先帮你识别目标用户、真实痛点和现有替代方案，再设计最小成本的验证动作，而不是直接让你做产品。
```

#### 场景 2：区分“喜欢”和“会付费”

```text
用户：很多朋友都说我的想法不错，这算验证成功吗？
AI：还不算。validate-idea 会重点帮你设计更强的验证信号，比如预约、预售、访谈和付费测试。
```

#### 场景 3：多个方向之间做选择

```text
用户：我有 3 个创业点子，不知道该先做哪个。
AI：适合用 validate-idea 先为每个方向设计最短验证路径，再比较哪一个更快接近真实付费需求。
```

## ✅ 适用场景

- **创业早期验证**：从 0 到 1 的方向判断
- **MVP 之前的风险控制**：避免过早投入开发
- **多想法筛选**：比较哪条路线更值得先试
- **用户访谈与需求验证设计**：建立更有效的问题清单和测试动作
- **付费意愿确认**：让验证从“口头兴趣”升级到真实承诺

---

## ❌ 不适用场景

- **已经有稳定用户和收入**：此时更适合做增长、留存或定价优化
- **纯技术实现问题**：它不是产品开发 Skill
- **需要大规模行业研究报告**：更适合作为创业实操框架，而不是正式咨询研究工具
- **已经确定要做，只差执行**：那更需要 MVP、交付与增长类 Skill

---

## ⚠️ 局限 & 风险

1. **验证通过不等于必然成功**：它只能提高方向正确率，不能保证商业结果
2. **创始人偏见仍会存在**：再好的问题清单，也可能被主观解释影响
3. **不同市场文化差异大**：《极简创业者》的方法更偏美国互联网创业语境
4. **过度验证也有代价**：如果无限拖延在验证阶段，也可能错过窗口
