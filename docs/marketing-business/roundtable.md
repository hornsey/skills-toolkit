# roundtable

> 多智能体辩论团 —— 通过 Scholar、Engineer、Muse 三类子 Agent 并行讨论，再由主 Agent 汇总结论

**Source:** [GitHub - robbyczgw-cla/roundtable](https://github.com/robbyczgw-cla/roundtable)

**Version:** 0.4.1 | **License:** 以仓库为准 | **Platforms:** 支持子 Agent / 多角色调度的 SKILL.md 兼容运行时

## 🎯 痛点 & 问题

面对复杂问题时，单个 Agent 往往有几个明显短板：

- **视角单一**：回答可能只偏技术、只偏研究，或只偏创意，缺少交叉校验
- **复杂决策容易拍脑袋**：商业分析、产品方向、技术选型往往需要多种思维方式共同评估
- **好点子经不起追问**：初看合理的建议，遇到质疑时缺少系统辩论过程
- **主 Agent 上下文压力大**：既要拆问题、又要分析、还要整合，容易顾此失彼

`roundtable` 的核心价值，是把“不同角色的思考碰撞”显式结构化，而不是假设一个模型能天然同时扮演所有专家。

## ⚙️ 原理

根据官方 SKILL.md，Roundtable 采用“主协调者 + 3 个并行角色 + 可选交叉质询”的结构。

### 核心角色

| 角色 | 职责 |
|------|------|
| **Captain / Coordinator** | 主 Agent，负责拆题、分配任务、控制轮次、汇总结论 |
| **Scholar** | 偏研究、事实、框架、证据与推理严谨性 |
| **Engineer** | 偏实现、约束、系统设计、可执行性 |
| **Muse** | 偏创意、叙事、替代方案与非直觉角度 |

### 核心机制

1. **并行第一轮讨论**
   - 主 Agent 将问题拆分后，同时交给 3 个不同角色分析
   - 减少顺序思考带来的路径依赖

2. **可选第二轮交叉质询**
   - 子 Agent 不只是各自输出，还可以互相挑战前一轮结论
   - 用来发现盲区、隐藏假设和不够稳固的推断

3. **模板化输出**
   - 不同角色可以使用各自模板或模型配置
   - 更适合稳定复用在研究、决策和高价值复杂任务中

4. **主 Agent 最终综合**
   - 最后仍由主 Agent 做合成
   - 重点不是“投票取平均值”，而是抽取冲突点、共识点与建议路径

### 工作流

```text
用户问题 → Captain 拆题并指定角色 → Scholar / Engineer / Muse 并行分析
→ （可选）第二轮交叉质询 → Captain 汇总最终建议
```

### 使用示例

#### 场景 1：商业决策

```text
用户：我们要不要进入教育行业做 AI Agent 产品？
AI：我会启动 roundtable，从市场研究、技术可行性和差异化叙事三个视角并行分析，再给你综合建议。
```

#### 场景 2：技术方案评估

```text
用户：我们该选多 Agent 架构还是单 Agent 工作流？
AI：我会让 Scholar 关注已知范式与证据，让 Engineer 评估复杂度与成本，让 Muse 提出更非传统的方案。
```

#### 场景 3：内容与品牌策略

```text
用户：如何把复杂的 AI 能力包装成市场容易理解的产品叙事？
AI：适合用 roundtable，让不同角色从事实、可落地性和品牌表达三个层面交叉验证。
```

### 何时值得启动 roundtable

当问题具有以下特征时更合适：

- 涉及多学科或多目标平衡
- 需要把“正确性”和“可执行性”同时考虑
- 用户希望看到不同观点之间的争论，而不是单结论输出
- 结果将影响产品、战略、投资或架构等高成本决策

---

## ✅ 适用场景

- **复杂商业决策**：市场进入、定价、产品方向、资源分配
- **技术架构评估**：多方案权衡、约束识别、实现优先级讨论
- **投资与研究分析**：需要证据、风险和故事线同时成立
- **高层汇报准备**：把单点想法打磨成更能经得起追问的论证
- **需要多视角创意探索**：既要创新，又不能脱离执行现实

---

## ❌ 不适用场景

- **简单事实查询**：一个搜索或直接回答就够，不需要多角色开会
- **低价值重复任务**：例行摘要、格式转换、简单翻译等无需此复杂度
- **极度紧急的即时动作**：如果目标只是迅速执行一个确定操作，roundtable 会增加时延
- **没有明确问题边界**：如果任务本身表述混乱，多角色并行只会放大噪声

---

## ⚠️ 局限 & 风险

1. **成本更高**：多角色并行通常意味着更高 token / API 成本
2. **响应更慢**：比单 Agent 输出多出拆题、协调和汇总步骤
3. **需要强协调**：若主 Agent 拆题不好，子角色输出再多也可能失焦
4. **并不自动更正确**：多视角可以减少盲点，但不能替代事实核验
5. **容易产出过长内容**：如果不设置边界，最终输出可能不够收敛
